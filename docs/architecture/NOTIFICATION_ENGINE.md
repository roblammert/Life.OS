# NOTIFICATION_ENGINE.md  
**Notification Engine Specification — Life.OS**  
*Part of the Life.OS Architecture Documentation Suite*

The Notification Engine is responsible for generating, scheduling, and delivering local, context‑aware notifications across Life.OS. Notifications are always private, always local, and never cloud‑pushed. They are powered by Life.Coach’s intelligence pipelines and surfaced through the Life.OS app shell.

This document defines the notification types, triggers, scheduling logic, UI surfaces, and integration with Life.Coach, the Life Graph, and the Sync Engine.

---

# 1. Purpose and Responsibilities

The Notification Engine provides **gentle, context‑aware prompts** that help users stay aligned with their goals, habits, and emotional well‑being.

### Core Responsibilities
- Generate notifications based on Life.Coach insights.
- Schedule notifications locally using browser APIs.
- Deliver notifications through the PWA interface.
- Respect user preferences and quiet hours.
- Provide actionable, non‑intrusive prompts.
- Integrate with all modules (Journal, Notes, Tasks, Storage).
- Never send data to external servers.

### Notification Philosophy
Notifications must be:
- **Helpful, not demanding**  
- **Contextual, not generic**  
- **Private, not cloud‑based**  
- **Optional, not required**  
- **Insightful, not noisy**

# 3. Notification Data Model

Notifications are stored locally in IndexedDB.

### `notifications`
| Field | Type | Description |
|-------|------|-------------|
| id | string | UUID |
| type | string | Notification type |
| title | string | Short title |
| body | string | Notification text |
| action | object | Optional action metadata |
| created_at | number | Timestamp |
| scheduled_for | number | Timestamp |
| delivered | boolean | Whether delivered |
| read | boolean | Whether user opened it |
| metadata | object | Module‑specific metadata |

### Example Notification
```
{ "id": "123", "type": "task_suggestion", "title": "You might want to revisit this task", "body": "Your energy is high this morning. Consider working on: 'Outline project proposal'.", "scheduled_for": 1712345678, "delivered": false, "read": false }
```

# 5. Scheduling Logic

Notifications are scheduled using:

- IndexedDB for persistence  
- Service worker for background delivery  
- Browser Notification API for display  

### Scheduling Rules
- Respect user quiet hours  
- Avoid notification bursts  
- Avoid repeating the same suggestion  
- Prioritize high‑value insights  
- De‑duplicate similar notifications  

### Quiet Hours
User‑configurable:
- Default: 10 PM – 7 AM  
- No notifications delivered during quiet hours  
- Non‑urgent notifications are queued

# 11. Error Handling

### Permission Denied
- Fall back to in‑app notifications  
- Prompt user to enable permissions  

### Service Worker Unavailable
- Deliver notifications only when app is open  

### Data Errors
- Skip invalid notifications  
- Log locally
